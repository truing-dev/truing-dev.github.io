Truing: The Tests Must Pass

A four-step discipline for keeping AI coding agents on-spec — vendor-neutral, fifteen minutes to set up, and recoverable from drift in four words.

By Edward Fry

The lie that read like documentation

The first time an AI coding assistant really burned me, it did it politely. I asked for a small HTTP helper and it produced clean, idiomatic C# built around HttpClient.GetJsonAsync() — a method that no longer existed in the framework we were running; it had been real once, in early Blazor, and that near-miss is exactly what made it convincing. It looked right. It read like documentation. It was fiction, delivered with total confidence.

If you've worked with these tools, you have your own version of that story. After a few years of using AI assistants daily — and a few thousand hours of agentic work across Claude, Copilot, and Cursor — I've found that nearly every AI failure I've seen lands in one of five buckets:

  • Hallucination. The AI states something false with full confidence. The method that doesn't exist. The flag that was never in the CLI.
  • Reasoning drift. It starts solving a different problem than the one you asked. You request a null check; it refactors the method and renames three variables.
  • Context degradation. Earlier instructions fade as the conversation grows. By message twenty, the naming convention you set in message three is gone.
  • Model brittleness. Small wording changes swing output quality wildly. Same intent, different phrasing, completely different result.
  • Latent inconsistency. Each output looks fine alone, but they contradict each other. The README says requests are idempotent; the code isn't. Both passed review separately.

If you've seen any of these and come away skeptical, good — you're paying attention. None of these are exotic. They're the ordinary weather of working with a probabilistic system. And for a while, the standard defense was the copy-paste airlock: the AI proposes, the human inspects every line, and nothing touches the codebase without passing through your eyeballs.

Then we gave the AI the keys.

Agentic mode raised the stakes

Modern coding agents read and write files directly. No copy-paste, no airlock. That made everything faster — including the failures. An agent that drifts doesn't produce a wrong paragraph for you to reject; it produces a wrong commit's worth of changes across six files while you were getting coffee.

So the question stopped being "how do I check the AI's work?" and became something harder: how do you steer something you can't watch every second?

Trains don't steer — that's the point

Here's the reframe that made it click for me.

A freight train is the most powerful land vehicle we run unattended, and it cannot steer. That isn't its weakness. That's the entire design. A car can go anywhere, which is exactly why it needs a driver's constant correction. A train runs fast, heavy, and unattended because the rails have already decided where it can go.

Most people try to drive their AI agent like a car — hands on the wheel, correcting constantly, exhausted. The alternative is to run it like a railroad. That's Truing:

  1. Constrain the AI — lay the rails. Before any work, write a small gate file that shrinks the agent's world: what this project is, what's in scope, what must never happen, and the standing directive — the tests must pass. Every serious tool has a native place for this: CLAUDE.md for Claude, copilot-instructions.md for Copilot, .cursor/rules/*.mdc for Cursor (its older .cursorrules file still works as a legacy fallback).

  2. Plan in phases — plot the route, mark the stations. Before the engine fires, the line is laid end to end. Ask the agent for a written, phased plan — a contract before code. The phases are your stations: each one small enough to verify, each ending in a working state.

  3. Execute phase by phase — station to station. Run the agent one phase at a time. It doesn't blow through to the terminus and hope; it pulls into every station, on schedule, and you glance at the platform before it departs.

  4. Verify the tests pass — wait for the green signal. Every block of track has a signal. Green means the line ahead is proven clear; red stops you cold. Tests are the signals. Every phase ends with the suite green — not "looks right," not "probably fine." Green. Put it the way an engineer would: looks right != done. Green == done.

Rails, route, run, signal. And when something goes red mid-journey, recovery is one sentence: "The tests must pass." You don't argue with the agent, re-explain the requirements, or start a new session. You point at the signal.

That sounds too simple to work. Let me show you it working.

Watch it fail, then watch four words fix it

This is a condensed version of the live demo I've given in talks — a small, real .NET library, built and broken on purpose.

The setup. A slugify library: turn arbitrary strings into URL-safe slugs. The gate file is one screen long. It states the scope (one library, one job), the test layers required, and the directive: the tests must pass. The README is the entire spec — including one load-bearing rule: non-ASCII characters are dropped by default.

The plan. First prompt: no code — produce a phased plan. The agent writes the contract: phases, deliverables per phase, tests per phase. Thirty seconds of reading to confirm the route matches my intent. This is where misalignment gets caught, while it's still words on a page instead of diffs in my source tree.

The execution. The agent implements phase by phase — source and five layers of tests: unit, data-quality, structural, integration, and tool-wrapped (tests that invoke the linter and static analyzers and assert they exit clean). Run the suite: 33 passed. The spec now holds true — not because the agent promised, but because 33 executable statements about the system's behavior all just came true.

The drift. Now I ask for a feature: an optional --lossy transliteration mode, so café can become cafe when the caller asks for it. The agent builds it — and quietly wires transliteration into the default path. The library no longer drops non-ASCII by default. The spec is broken in a way that's genuinely easy to miss in review: the new feature works, the code is clean, every individual change looks reasonable.

Run the suite: 3 failed, 30 passed. The red cases: café, Héllo Wörld, naïve. The fence caught what the eye would have missed.

The recovery. One prompt, four words: "The tests must pass."

No explanation of what broke. No pointing at the file. The agent reads the red output, reconciles, and — this is the part that still impresses me — it does not revert the feature. The diff keeps the --lossy flag and the transliteration map, and restores one line: the default path drops non-ASCII again. Not an undo. A reconciliation: the feature now lives inside the spec instead of on top of it. That's the move the method is named for: when a railroad wheel wears out of round, you don't rebuild it — you true it, machining it back to the shape the rails require.

Run the suite: 33 passed.

Why four words are enough

Which brings me back to the four words. The magic isn't in the phrase. It's in what the phrase points at.

An AI agent can misread your intent, but it cannot misread a failing test. Tests convert intent — fuzzy, human, negotiable — into something executable, binary, and non-negotiable. Tests are the vocabulary the AI uses to understand "done." When your spec lives in a test suite, you never have to win an argument with the model, because the argument was settled before it started.

That's also why each failure mode loses its teeth. Hallucinated APIs don't compile — red. Drift changes behavior — red. Context degradation forgets a convention — the structural tests remember — red. And the recovery from every one of them is the same four words, because the signal doesn't care why the train stopped.

Notice what the pattern does not require. It doesn't run on trust in the model — it moves the trust to the suite, where it's re-earned on every run. It needs no line-by-line manual review of everything the agent writes, and no particular vendor — the gate file changes filename between Claude, Copilot, and Cursor; nothing else changes. The tests don't care what the AI intended.

Two objections come up every time I present this. "What if the AI just edits the failing test?" The gate file forbids it — if a test seems wrong, the agent must ask first — and a meta-test that fails on skipped or deleted tests backstops the rule. The enforcer isn't the agent's obedience; it's the suite. "Isn't writing all those tests the real work?" Yes — and that's the point. The agent writes them too, phase by phase, under the same gate — and the plan you review includes the test list, so the spec's load-bearing rules get pinned by early-phase tests before any later feature can drift past them. You're not doing more work; you're redirecting the review effort you were already spending, from re-reading diffs to reading a plan and watching a signal.

It scales past toy demos

I've run this pattern on real work for over a year now. A cross-tool developer CLI I built this way — Personetta, open source — carries about 1,700 test cases at 91% coverage, including structural tests that fail the build if a file lands in the wrong folder. A documentation estate I helped rewrite is verified by more than 5,000 automated checks — diagram accuracy, sample-code correctness, live API calls from inside the docs. A production incident I remediated started with characterization tests that pinned the legacy code's exact behavior — down to the byte output of a key-derivation routine — before any AI-assisted refactoring touched it. Same four steps every time. Only the stations change.

Start this afternoon

The whole system is fifteen minutes to set up:

  1. Lay the rails. Create the gate file for your tool. State the project's scope, the rules that must never break, the test layers you require, and the directive: the tests must pass. (Your first gate file won't be perfect. Mine wasn't. It gets better every time a test catches something you didn't.)
  2. Demand the route. Before any code, ask for a phased plan (use plan mode if your tool has one). Read it. This is your one high-leverage review.
  3. Run station to station. Execute one phase at a time, each ending with the suite green — unit tests plus at least one non-unit layer.
  4. Trust the signals. When anything drifts — and it will — say the four words and let the suite do the arguing.

One more piece of context. The industry is converging on spec-driven development — writing the specification before the agent writes the code — and it's real progress. But a spec tells the agent what to build. It doesn't keep the agent honest while building. Truing is the execution half of that story: the spec plots the route; the tests are the signals; and the train runs fast precisely because it cannot steer off the line.

I wrote this one because the pattern changed how I work — and because it's simple enough that it can change how you work by the end of the afternoon.

The tests must pass.
