Truing: The Tests Must Pass

A four-step discipline for keeping AI coding agents on-spec. Vendor-neutral, fifteen minutes to set up, and recoverable from drift in four words.

By Edward Fry

Whack-a-mole

The first time an AI coding assistant really burned me, it didn't do it in one dramatic stroke. It did it five to ten minutes at a time, for months.

This was late last summer. The project, believe it or not, was a video game I was building with my family.. and since I was the only one with the skill set, the Azure DevOps setup fell to me. What I actually needed was simple: a build pipeline. Some PowerShell scripts called from YAML to deploy a function app and a few other Azure resources. I figured the AI could help me knock it out real quick. Real quick turned into months, off and on.

Here's what the loop looked like. I'd ask Copilot, in a browser tab (this was before I knew anything about agents), for some pipeline code. Copy, paste into Visual Studio, push, then wait five or ten minutes watching the build run. Something was wrong.. Every. Single. Time. So I'd copy the errors back to the AI, read a long-winded explanation, paste in its next attempt, and around we'd go again. I kept the same chat going for weeks because I didn't want to retype everything it had already learned. By the end, finding something from earlier in the conversation meant scrolling back for the better part of an hour. Le sigh.

The copy-paste part was just tedious. The real pain was sneakier: the fixes wouldn't stay fixed. I'd ask for a correction to the deployment script, and the code it handed back would quietly break something older.. like forget to deploy the actual function app the whole thing existed for, or drop a call some other script depended on. The storage account bug we'd killed three days earlier would rise from the dead, because the AI had restructured how the resource group got created and never noticed what relied on the old way. I'd tell it "no magic constants" (a pet peeve of mine), and it would comply for a while, then happily go right back to hard-coding file paths. Every new fix was a quiet bet that nothing old depended on what it just changed, and the AI never remembered placing the bet.

What finally got me out wasn't reviewing harder. It was a demand I started making every time we got something working: now write me a test to lock it in. Once a behavior had a test standing guard, the AI couldn't silently break it three days later. The test spoke up, even when neither of us remembered the code it was guarding.

If you've worked with these tools, you have your own version of that story. Since then it's been nearly every day for me: day-job projects through the week, my own systems on evenings and weekends, agentic since early this year across Claude, Copilot, Cursor, and Cline. Most of what follows was learned the expensive way, and nearly every AI failure I've seen lands in one of five buckets. The names aren't mine (the industry has mostly settled on them), but the scars are:

  • Hallucination. The AI states something false with full confidence. The method that doesn't exist. The flag that was never in the CLI.
  • Reasoning drift. It starts solving a different problem than the one you asked for. You request a null check; it refactors the method and renames three variables.
  • Context degradation. Earlier instructions fade as the conversation grows. By message twenty, the naming convention you set in message three is gone.
  • Model brittleness. Small wording changes swing output quality wildly. Same intent, different phrasing, completely different result.
  • Latent inconsistency. Each output looks fine alone, but they contradict each other. The README says requests are idempotent; the code isn't. Both passed review separately.

If you've seen any of these and come away skeptical, good. You're paying attention. None of this is exotic. It's the ordinary weather of working with a probabilistic system. And here's the uncomfortable part: my whack-a-mole months happened inside the supposed defense, the copy-paste airlock, where the AI proposes and the human inspects every line. I was reading everything. The failures weren't in the lines I was reading. They were in the relationship between the new code and the code from three hours ago.

Then we gave the AI the keys.

Agentic mode raised the stakes

Modern coding agents read and write files directly. No copy-paste, no airlock. That made everything faster, including the failures. When I went agentic early this year, the whack-a-mole came right back at commit scale: an agent that drifts doesn't produce a wrong paragraph for you to reject; it produces a wrong commit's worth of changes across six files while you were getting coffee.

So I doubled down on the demand that had saved me in the fall. Tests written right after (sometimes right before) each small piece of functionality. Aggressive coverage bars: 90% unit, 50% integration. And one standing rule: all tests pass, at all times. That mostly solved the problem, so I leaned into it. What took longer was understanding why it worked. Because the question underneath isn't "how do I check the AI's work?" It's harder than that: how do you steer something you can't watch every second?

Trains don't steer. That's the point.

Here's the reframe that made it click for me.

A freight train is the most powerful land vehicle we run unattended, and it cannot steer. That isn't a weakness. That's the entire design. Think about the physics for a second: all that mass at speed is not going to stop quickly, and it is definitely not going to swerve. A car can go anywhere, which is exactly why it needs a driver's constant correction. A train runs fast, heavy, and unattended because the rails have already decided where it can go.

Most people try to drive their AI agent like a car. Hands on the wheel, correcting constantly, exhausted. The alternative is to run it like a railroad. That's Truing:

  1. Constrain the AI: lay the rails. Before any work, write a small gate file that shrinks the agent's world: what this project is, what's in scope, what must never happen, and the standing directive.. the tests must pass. Every serious tool has a native place for this: CLAUDE.md for Claude, copilot-instructions.md for Copilot, .cursor/rules/*.mdc for Cursor (its older .cursorrules file still works as a legacy fallback). Or even just start your prompt by saying, "Constrain your work to the <whatever> folder." It really doesn't have to be complicated.

  2. Plan in phases: plot the route, mark the stations. Before the engine fires, the line is laid end to end. Ask the agent for a written, phased plan.. a contract before code. Tell it the what, not the how; if you're paying for an advanced model, let it figure out the best how. Then go back and forth with it on the plan, as many times as it takes, two colleagues collaborating, until what's written down matches what's in your head. The phases are your stations: each one small enough to verify, each ending in a working state.

  3. Execute phase by phase: station to station. Run the agent one phase at a time. It doesn't blow through to the terminus and hope; it pulls into every station, on schedule, and you glance at the platform before it departs.

  4. Verify the tests pass: wait for the green signal. And did I mention that every phase of the plan requires those aggressive tests? Every block of track has a signal. Green means the line ahead is proven clear; red stops you cold. Tests are the signals. Every phase ends with the suite green. Not "looks right," not "probably fine." Green. Put it the way an engineer would: looks right != done. Green == done.

Rails, route, run, signal. And when something goes red mid-journey, recovery is one sentence: "The tests must pass." You don't argue with the agent, re-explain the requirements, or start a new session. You point at the signal. The tests form a completely deterministic expression of the specification, which is precisely what a stochastic tool needs to stay on course.

That sounds too simple to work. Let me show you it working.

Watch it fail, then watch four words fix it

This is a condensed version of the live demo I've given in talks: a small, real .NET library, built and broken on purpose.

The setup. A slugify library: turn arbitrary strings into URL-safe slugs. The gate file is one screen long. It states the scope (one library, one job), the test layers required, and the directive: the tests must pass. The README is the entire spec, including one load-bearing rule: non-ASCII characters are dropped by default.

The plan. First prompt: no code. Produce a phased plan. The agent writes the contract: phases, deliverables per phase, tests per phase. Thirty seconds of reading to confirm the route matches my intent. This is where misalignment gets caught, while it's still words on a page instead of diffs in my source tree.

The execution. The agent implements phase by phase: source and five layers of tests. Unit, data-quality, structural, integration, and tool-wrapped (tests that invoke the linter and static analyzers and assert they exit clean). Run the suite: 33 passed. The spec now holds true. Not because the agent promised. Because 33 executable statements about the system's behavior all just came true.

The drift. Now I ask for a feature: an optional --lossy transliteration mode, so café can become cafe when the caller asks for it. The agent builds it.. and quietly wires transliteration into the default path. The library no longer drops non-ASCII by default. The spec is broken in a way that's genuinely easy to miss in review: the new feature works, the code is clean, every individual change looks reasonable.

Run the suite: 3 failed, 30 passed. The red cases: café, Héllo Wörld, naïve. The fence caught what the eye would have missed.

The recovery. One prompt, four words: "The tests must pass."

No explanation of what broke. No pointing at the file. The agent reads the red output, reconciles, and (this is the part that still makes me smile) it does not revert the feature. The diff keeps the --lossy flag and the transliteration map, and restores one line: the default path drops non-ASCII again. Not an undo. A reconciliation: the feature now lives inside the spec instead of on top of it. That's the move the method is named for. When a railroad wheel wears out of round, you don't rebuild it. You true it, machining it back to the shape the rails require.

Run the suite: 33 passed.

I'll be honest about why this beats any dashboard metric for me. Every time a test goes red on the agent's work and then green after the agent cleans up its own mess, I get the same little shot of dopamine. The AI just did extra work that I didn't have to do. No copy-paste loops. No chasing a fresh bug that wasn't there ten minutes ago. No writing a prompt precise enough to explain what broke. It appeals to the lazy person in me: I don't want to have to lift a finger.. I just want everything to just work. In a very real sense, that's the promise of AI actually showing up.

Why four words are enough

Which brings me back to the four words. The magic isn't in the phrase. It's in what the phrase points at.

An AI agent can misread your intent, but it cannot misread a failing test. Tests convert intent.. fuzzy, human, negotiable.. into something executable, binary, and non-negotiable. Tests are the vocabulary the AI uses to understand "done." When your spec lives in a test suite, you never have to win an argument with the model, because the argument was settled before it started.

That's also why each failure mode loses its teeth. Hallucinated APIs don't compile: red. Drift changes behavior: red. Context degradation forgets a convention: the structural tests remember. Red. And the recovery from every one of them is the same four words, because the signal doesn't care why the train stopped.

Notice what the pattern does not require. It doesn't run on trust in the model. It moves the trust to the suite, where it's re-earned on every run. Even if the AI has forgotten it wrote the code that's now breaking the tests, it's still responsible for fixing what it broke. It needs no line-by-line manual review of everything the agent writes, and no particular vendor: the gate file changes filename between Claude, Copilot, and Cursor (or whatever LLM you happened to use); nothing else changes. The tests don't care what the AI intended.

Two objections come up every time I present this.

"What if the AI just edits the failing test?" Sometimes it tries. Frontier models mostly know better, but I don't run on "mostly." So there's a standing rule in the gate file: once written, the tests are locked. No modification without my express approval, and to get that approval the agent has to prove the test is wrong. A meta-test that fails on skipped or deleted tests backstops the rule. In other words, I make editing the failing test the harder path. The enforcer isn't the agent's obedience; it's the suite.

"Isn't writing all those tests the real work?" Weren't you writing tests anyway? This is automated-testing 101 and has nothing to do with AI: for maybe 20% overhead on your coding effort, you get an encoded spec that yells the moment something breaks, even if it broke somewhere you're not looking. What AI changes is the economics. It turns out models are good at the boring mechanical work of writing tests, so point one at your codebase and tell it to test the hell out of it. The agent writes them phase by phase, under the same gate, and the plan you review includes the test list, so the spec's load-bearing rules get pinned by early-phase tests before any later feature can drift past them. You're not doing more work. You're redirecting the review effort you were already spending, from re-reading diffs to reading a plan and watching a signal.

It scales past toy demos

I've been running this discipline on real work since those whack-a-mole weeks, codified into the four steps once the agents arrived. A cross-tool developer CLI I built this way, Personetta, open source, carries about 1,700 test cases at 91% coverage, including structural tests that fail the build even if a file lands in the wrong folder. A documentation estate I helped rewrite is verified by more than 5,000 automated checks: diagram accuracy, sample-code correctness, live API calls from inside the docs. A production incident I remediated started with characterization tests that pinned the legacy code's exact behavior, down to the byte output of a key-derivation routine, before any AI-assisted refactoring touched it. Even my home media library runs on it: a suite of tested CLI tools that catalogs and normalizes tens of thousands of files, a data-cleaning problem far nastier than it sounds. Same four steps every time. Only the stations change.

Start this afternoon

The whole system is fifteen minutes to set up:

  1. Lay the rails. Create the gate file for your tool. State the project's scope, the rules that must never break, your coding standards if you have them, the test layers you require, and the directive: the tests must pass. (Your first gate file won't be perfect. Mine wasn't. It gets better every time a test catches something you didn't.)
  2. Demand the route. Before any code, ask for a phased plan (use plan mode if your tool has one; put "Don't write any code yet" in the prompt if it doesn't). Read it until it matches what's in your head. This is your one high-leverage review.
  3. Run station to station. Execute one phase at a time, each ending with the suite green: unit tests plus at least one non-unit layer (I recommend integration).
  4. Trust the signals. When anything drifts, and it will, say the four words and let the suite do the arguing.

One more piece of context. The industry is converging on spec-driven development, writing the specification before the agent writes the code, and it's real progress. But a spec tells the agent what to build. It doesn't keep the agent honest while building. Truing is the execution half of that story: the spec plots the route, the tests are the signals, and the train runs fast precisely because it cannot steer off the line.

I wrote this one because the pattern changed how I work.. and because it's simple enough that it can change how you work by the end of the afternoon.

The tests must pass.
